TIER 2Lesson 0005 · 약 30분

데이터 plumbing과 INCI 정규화

상위 모든 tier의 GIGO를 결정한다. "garbage in, garbage out" — 95% acceptance는 입력 정규화 90%에서 절반이 결정난다.

이 레슨의 약속 마치면 (1) CosIng·ECHA·IFRA·내부 R&D 4개 소스의 데이터 모델과 fetch 전략을 알고, (2) INCI 정규화의 7가지 실패 모드를 식별·방어하며, (3) Substance Identity Hub 패턴으로 모든 후속 tier가 같은 "성분 ID"를 가리키게 만든다. ToxTool이 90% 성분 커버한다는 표현 뒤에 숨은 노가다를 직접 본다.

1. 문제 정의 — 왜 plumbing이 이토록 어려운가

"Niacinamide 4%"는 입력이지만, 시스템 안에서는 다음 매핑을 모두 풀어야 한다:

2. INCI 정규화의 7가지 실패 모드

#실패예방어
1대소문자·공백 변종"Niacinamide" vs "NIACINAMIDE" vs "niacinamide " (trailing space)강제 normalize: trim + Title Case 또는 lowercase canonical key
2구두점·기호 변종"Acrylates/C10-30 Alkyl Acrylate Crosspolymer"의 /, -, 띄어쓰기regex 정규화 사전 + 알려진 변종 매핑 테이블
3동의어·구명칭"Sodium PCA" vs "PCA Sodium", "Tocopherol" vs "Vitamin E" (라벨용은 INCI 강제)synonym 사전 (CosIng "Other Names" 활용) + LLM fallback (확신도 표시)
4한국명·중문명처방표가 한국 표준명으로 작성된 경우 — "나이아신아마이드"한국 식약처 명명 ↔ INCI 매핑 테이블. 식약처 화장품 성분명 사전 참조.
5염·수화물·이성질체"Sodium Hyaluronate"는 "Hyaluronic Acid"의 sodium salt — 평가 시 같은 toxicology를 쓰지만 분자량·노출량은 다름"parent compound" 필드 + salt/hydrate flag. NOAEL은 parent에서 가져오되 SED는 실제 분자량으로 계산.
6혼합물·복합 성분"Phenoxyethanol (and) Ethylhexylglycerin"처럼 복합 보존제 (벤더 제품) — 처방표에 한 줄로 표기되나 INCI는 2개처방 입력 단계 검증: (and) 또는 + 토큰 발견 시 분할 강제. 비율 미정이면 HITL.
7알려지지 않은 신성분"Centella Asiatica Reservatrol Conjugate"(가공) — CosIng·ECHA·CIR·SCCS 어디에도 없음fallback chain 전부 실패 → HITL. 침묵 fabrication 절대 금지.
CosIng 검색의 함정 CosIng는 공식 API가 없다. UI 검색은 fuzzy하게 동작하지만, 시스템에서 자동화하려면 (a) 정기 dump 활용, (b) HTML 스크래핑, (c) 자체 사전 구축 중 선택. 공식 표기는 "하나의 INCI에 대응하는 CosIng 레코드는 하나"인데, 동의어 검색이 매우 관대해서 잘못된 entry를 끌어올 수 있다. fuzzy match는 confidence score를 강제하고 0.95 미만은 HITL.

3. 주요 데이터 소스 4개

① CosIng — EU 화장품 성분 데이터베이스

제공 데이터
INCI 정규명, 기능(SMOOTHING·HUMECTANT 등), CAS/EC, EU Reg Annex II~VI 등재 여부, 제한 조건(농도·라벨), "Other Names" 동의어
API
공식 API 없음. 공개 검색 UI만. 자동화 옵션: (1) HTML scraping (rate-limit 주의), (2) EU OpenData portal에서 dump 받기, (3) 사내 캐시 사전 구축
갱신 빈도
EU Reg 개정 시 (실무상 분기~반기). 변경 감지가 regulation alert의 핵심.
활용 우선순위
1순위. INCI normalization, Annex 제약, 기능 라벨 모두 여기서.
실패 패턴
"INCI 정확히 일치하지 않으면 0건" → fuzzy match 자체 구현 필요. nano 표기(예: "Titanium Dioxide (nano)") 별도 entry.

② ECHA REACH — 화학물질 등록 DB

제공 데이터
등록 substance의 정체(CAS/EC/IUPAC), 물리화학적 특성, 독성 데이터 요약(NOAEL, LD50, irritation, sensitisation), classification & labelling
API
OData API + bulk export 제공 (echa.europa.eu). 일부 detail은 robot policy 확인 필요. 화장품 컨텍스트가 아니라 화학물질 컨텍스트 — 노출 시나리오는 본인이 매핑.
활용 우선순위
SCCS Opinion 없는 성분의 NOAEL fallback 1순위. CAS-key로 정확 매칭.
실패 패턴
(1) 화장품 노출(dermal)과 REACH 시험(주로 oral) 경로 차이 → SCCS NoG 12판 §3-6.X에 따라 보정. (2) 같은 substance에 여러 dossier가 있을 수 있음 → 최신·가장 완전한 것 선택.

③ IFRA Standards — 향료 알레르겐·한도

제공 데이터
49 IFRA Standards (Prohibition, Restriction, Specification). 알레르겐 24+ + EU Reg 알레르겐 26종. 제품 카테고리(IFRA 11 카테고리)별 최대 사용량.
API
공식 API 없음. PDF·Excel 다운로드 + 라이브러리 멤버십. 라이선스 주의 — 사내 사용은 가능하지만 redistribution 제한.
활용 우선순위
처방에 fragrance 또는 essential oil이 있으면 필수. leave-on 카테고리(IFRA 5A·5B 등)와 처방의 사용량 비교.
실패 패턴
"Parfum" 한 줄에 수십 개 향료 성분 → 향료 제조사의 알레르겐 declaration 별도 fetch (XML/PDF 형식). EU Reg 26 declared allergens는 라벨 의무이므로 합산이 0.01% leave-on / 0.1% rinse-off 임계 초과 여부 별도 계산.

④ 내부 R&D / Formulation / LIMS

제공 데이터
처방표(INCI %, 벤더 raw material code, 함량 tolerance), 안정성 시험 결과(가속·장기), 미생물 challenge test 결과, HRIPT/임상 결과, 부작용 보고(이미 시판된 경우)
API
회사마다 다름 — ERP/PLM/LIMS의 SQL 직접 접근 또는 export 파일. 이 매핑이 in-house 시스템의 최대 강점. 외부 SaaS(ToxTool·Worldover)는 이 데이터에 직접 접근 못한다.
활용 우선순위
CPSR Part A.2·A.3·A.4·A.9·A.10의 거의 전부. 외부 데이터 위에 사내 데이터를 덮어쓰지 않음 — 같은 처방의 안정성은 사내 데이터가 정답.
실패 패턴
(1) 처방 변경 → CPSR 자동 재생성 트리거 없음(처방 시스템과 CPSR 시스템 연결이 안 됨). (2) HRIPT 결과가 PDF 스캔으로만 존재 → OCR + 구조화. (3) 시판 부작용 데이터의 RA 보고 의무(SUE)와 일반 UE 구분.

4. Substance Identity Hub — 통합 패턴

네 소스에서 들어오는 데이터를 어떻게 한 줄로 묶는가. 중심에 INCI를 두면 안 된다 (변종이 너무 많다). 중심은 substance_id (내부 UUID):

from pydantic import BaseModel
from typing import Literal, Optional

class SubstanceIdentity(BaseModel):
    substance_id: str  # 내부 UUID — 절대 변하지 않음
    inci_canonical: str  # CosIng 기준 정규 INCI
    inci_synonyms: list[str]  # 모든 알려진 변종(키 검색용)
    cas: Optional[str]
    ec: Optional[str]
    korean_name: Optional[str]
    iupac_name: Optional[str]
    parent_compound_id: Optional[str] = None  # 염·수화물의 모분자

class CosingRecord(BaseModel):
    substance_id: str  # FK
    functions: list[str]
    annex_ii_listed: bool  # 금지
    annex_iii_restrictions: list[Restriction] = []  # 제한
    annex_iv_color: Optional[ColorAnnex] = None
    annex_v_preservative: Optional[PreservativeAnnex] = None
    annex_vi_uv_filter: Optional[UvFilterAnnex] = None
    last_fetched: datetime  # 캐시 freshness

class EchaRecord(BaseModel):
    substance_id: str  # FK
    dossier_url: str
    physchem: PhysChemProperties
    tox_summary: ToxSummary  # NOAEL·LD50 등

class InternalRdRecord(BaseModel):
    substance_id: str  # FK
    raw_material_code: str  # 사내 코드
    vendor: str
    purity_spec: str
    impurity_data: Optional[ImpurityData] = None

# 검색은 inci_canonical 또는 inci_synonyms 어디로 들어와도 같은 substance_id로 귀결
def resolve_substance(raw_inci: str) -> tuple[SubstanceIdentity, float]:
    """Returns (identity, confidence)."""
    normalized = normalize(raw_inci)  # trim, case, punct
    if hit := exact_match(normalized):
        return hit, 1.0
    if hit := synonym_lookup(normalized):
        return hit, 0.99
    if hit := fuzzy_match(normalized, threshold=0.95):
        return hit, hit.score
    # LLM fallback — 알려진 변종/오타 추정
    if hit := llm_propose(normalized):
        return hit, 0.7  # 자동 통과 X — HITL 검토
    raise SubstanceNotResolvedError(normalized)
왜 substance_id가 중심인가 CosIng가 INCI 표기를 바꿔도(예: "Glycerin" → "Glycerine" 표준화), 내부 substance_id는 안 변한다. 모든 후속 데이터(NOAEL extraction 결과, MoS 계산 결과, 과거 CPSR 작성 이력)가 substance_id로 묶여있어 INCI 변경에 무너지지 않는다. 외부 ID(INCI)는 변할 수 있다 — 내부 ID는 immutable.

5. 캐싱·freshness·regulation alert

네 소스는 갱신 빈도가 다르다. 캐시 전략을 각각:

소스TTL알림 트리거
CosIng (Annex 변경)1주"기존 자유 사용 → Annex III 제한 추가" 발생 시 해당 성분 사용 중인 모든 제품의 CPSR 재평가 알림. Worldover의 핵심 가치 패턴.
ECHA REACH1개월NOAEL revision은 매우 드물지만 발생 시 영향 큼. 변경 감지 → MoS 재계산 큐.
IFRA Standards매 amendment 발표 시amendment 발표 → 알레르겐·한도 변경 → 향료 함유 제품 일괄 재평가.
내부 R&D실시간 (이벤트)처방 변경 → CPSR 즉시 dirty 표시 → SA 재검토 queue.
"variant invalidation"의 마법 Substance Identity Hub가 잘 만들어져 있으면, "phenoxyethanol Annex V 농도 한도가 1.0% → 0.7%로 강화"라는 단일 이벤트가 (a) 어느 성분이 영향받는지, (b) 어느 제품이 영향받는지, (c) 어느 CPSR이 dirty인지를 한 쿼리로 답한다. 사내 도구의 가장 가치 있는 자동화.

6. 처방 입력 검증 — 마지막 방어선

Tier 2의 입구에서 시스템이 강제해야 할 검증:

def validate_formulation(formulation: list[FormulationLine]) -> ValidationReport:
    issues = []

    # 1. 모든 INCI가 substance_id로 resolve되는가
    for line in formulation:
        identity, conf = resolve_substance(line.raw_inci)
        if conf < 0.95:
            issues.append(LowConfidenceSubstance(line, conf))

    # 2. 농도 합 ≈ 100%
    total = sum(line.concentration_pct for line in formulation)
    if not (99.5 <= total <= 100.5):
        issues.append(ConcentrationSumOutOfRange(total))

    # 3. "(and)" 또는 "+" 토큰 발견 시 분할 강제
    for line in formulation:
        if "(and)" in line.raw_inci or re.search(r"\s\+\s", line.raw_inci):
            issues.append(MultiSubstanceLineNeedsSplit(line))

    # 4. Annex II(금지) 성분 사용 시 즉시 fail
    for line in formulation:
        identity, _ = resolve_substance(line.raw_inci)
        if cosing_get(identity.substance_id).annex_ii_listed:
            issues.append(ProhibitedSubstance(identity))

    # 5. Annex III/V/VI 제한 농도 초과
    for line in formulation:
        identity, _ = resolve_substance(line.raw_inci)
        cos = cosing_get(identity.substance_id)
        if over_limit(line.concentration_pct, cos):
            issues.append(ConcentrationOverAnnexLimit(line, cos))

    # 6. 알레르겐 26 / 24 합산 한도
    allergen_sum = sum_allergens(formulation)
    if allergen_sum > 0.001:  # 0.1% 또는 leave-on 0.001%
        issues.append(AllergenLabelingRequired(allergen_sum))

    return ValidationReport(issues=issues)

이 6개가 통과되어야 Tier 1·3·4로 진입. 입력 검증이 95% acceptance의 가장 큰 단일 lever. SA가 검토 단계에서 "이거 농도 합이 안 맞는데?"를 발견하면 그 시점부터 시스템 신뢰가 0이 된다.

7. Quiz — 4문항

Q1. 처방표의 한 줄이 "Sodium Hyaluronate"이다. NOAEL은 SCCS Opinion의 "Hyaluronic Acid" entry에서 가져왔다. MoS 계산 시 두 분자의 분자량 차이(~22 Da)를 어떻게 처리할까?

답 보기
두 가지 옵션. (a) 분자량 비율 보정: parent 분자(Hyaluronic Acid)의 NOAEL을 그대로 사용하되, salt 형태의 실제 노출량을 분자량 비율로 보정. Hyaluronic Acid MW vs Sodium Hyaluronate MW의 비율로 SED를 보정해 "유리 산 기준" SED 산출. 보정 인자가 작은 분자(~3%)면 보통 무시 가능 — 명시는 함. (b) 보수적 접근: parent NOAEL 그대로 + salt 분자량 보정 안 함 — 결과적으로 약간 보수적 (SED 조금 과대평가). 큰 차이 없으면 권장.
중요한 건 시스템이 parent-salt 관계를 알고 있고 narrative에 명시한다는 것. SA가 "왜 Sodium Hyaluronate에 Hyaluronic Acid NOAEL을 썼나"를 보고서에서 찾을 수 있어야 함. Substance Identity Hub의 parent_compound_id 필드가 이걸 가능하게 한다.

Q2. CosIng 검색에서 "Glycerin"을 입력했더니 (a) "Glycerin"과 (b) "Glycereth-26" 두 entry가 fuzzy hit으로 떴다. confidence는 (a) 0.97, (b) 0.93. 시스템은 어떻게 행동해야 하는가?

답 보기
(a)를 선택 + 사용자에게 확정 알림 또는 silent log. 임계 0.95 위는 자동 통과지만, 0.97과 0.93 사이가 좁고 둘 다 hit이므로 second-best의 정체를 audit log에 남긴다. 동시에 (a)와 (b)의 INCI 표기가 완전히 다른 분자(glycerin은 모노머, glycereth-26은 PEG 유도체)임을 인지 — 단순 substring 유사도가 아니라 화학 동일성이 기준이어야 한다.
방어 강화: fuzzy match에 "CAS 일치"를 우선 룰로. CAS가 함께 있으면 fuzzy match를 건너뛰고 exact CAS lookup. 처방표가 CAS 없이 INCI만 들고 오면 fuzzy 위험 영역. → R&D에 "CAS 병기" 요청을 plumbing 1차 개선과제로.

Q3. 매월 CosIng 동기화를 돌리는데, 어느 날 "Methylisothiazolinone"의 Annex V 한도가 0.01% → 0.0015% (leave-on)로 강화되었다. 시스템은 다음 액션을 자동으로 트리거해야 한다. 무엇무엇인가?

답 보기
① 영향 제품 식별: WHERE formulation contains substance_id(MIT) 쿼리 → leave-on 제품 중 농도 ≥ 0.0015% 모두 list. 결과 표를 SA·RA에 알림. ② 해당 CPSR을 "dirty" 표시: 자동 재평가 큐에 등록 (수동 처리 우선순위 P0). ③ 처방 변경 제안: Tier 1로 SED 재계산, MoS 재계산, MoS < 100 또는 농도 한도 위반 case는 자동 분리. ④ regulation change log: amendment URL, 적용일, 유예기간 기록 — Tier 4 reasoning이 narrative에 그대로 인용. ⑤ 경쟁사·시장 모니터링: 같은 변경에 다른 제조사가 어떻게 대응했는지 추적 (선택).
이 5단계가 외부 SaaS가 못 하는 영역 — 우리 사내 데이터에 직접 붙어있기 때문에 가능. Worldover가 그들의 "alert" 가치로 마케팅하는 그 자리.

Q4. R&D에서 새 처방표를 던졌는데, 한 줄에 "Centella Asiatica Reservatrol Conjugate"가 있다. CosIng·ECHA·CIR·SCCS 어디에도 없다. 시스템은 어디서 멈춰야 하며, SA에게 무엇을 전달해야 하는가?

답 보기
Tier 2 입력 검증 단계에서 즉시 중단. 진행 X. SA에게 전달할 payload:
- 처방 line + raw INCI
- 시도한 fallback chain: CosIng (miss), ECHA (miss), CIR (miss), SCCS Opinion search (miss), EFSA (miss), PubChem CAS 추정 (miss)
- 추측: "구조상 Centella 추출 + 레스베라트롤 컨주게이트로 보임. 모분자(레스베라트롤)의 NOAEL은 EFSA에 있음. SA 판단: (a) read-across로 모분자 NOAEL 적용 정당화 가능한가?, (b) 자체 시험 필요한가?, (c) 처방 교체?"
- SA 결정 기록 슬롯
이 흐름 자체가 시스템의 가치 — "모르겠다"를 명시적·구조적으로 노출. 자동 추측·생성 절대 금지. SA의 결정이 다시 시스템 학습 데이터로 흡수되어 다음 신성분 case에 활용.
