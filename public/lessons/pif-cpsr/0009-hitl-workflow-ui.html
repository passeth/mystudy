HITL Workflow UI 설계 — SA 의사결정의 시스템 흡수

법적 책임은 영구히 사람 SA에 있다. UI의 일은 SA의 시간을 적게 쓰고, 결정을 분명히 기록하고, 결정이 다음 case의 룰북이 되게 만드는 것. "예쁜 dashboard"가 아니라 decision pipeline.

이 레슨의 약속 마치면 (1) 5개 화면(Queue·Case Detail·Decision Recorder·Audit Log·Style Anchor Editor)의 정보 구조, (2) 각 화면이 어떤 SA 행동을 한 클릭으로 끝내야 하는지, (3) 시스템이 SA 결정을 어떻게 학습 자산으로 변환하는지, (4) 솔로 메이커에게 적합한 thin UI stack(Next.js/Streamlit/Notion·Linear vs full custom)의 선택 기준. 가장 큰 함정은 "general purpose dashboard"다. SA가 하루 5분만 쓰고도 시스템이 전진하는 UI가 목표.

1. SA의 하루 — workflow에서 출발

UI 설계 전에 SA가 시스템과 상호작용하는 6개 시나리오를 먼저 정의:

#시나리오발생 빈도 (예상)SA 시간 예산
S1새 처방의 draft 검토 → 서명 또는 수정주 2~5건건당 5~15분
S2P0 HITL 알림 (MoS < 임계)주 0~1건건당 20~30분
S3P1 HITL 알림 (NOAEL 부재, evaluator 3회 실패 등)주 1~3건건당 10~20분
S4처방 변경 알림 (Tier 2 detected)주 1~2건건당 5분 (RA 판단)
S5Style anchor 갱신 review월 1회30분
S6분기 audit (감독 대응)분기 1회2시간
SA의 시간 예산 = UI의 가장 강한 제약 SA는 RA·QA·외부 컨설팅 등 다른 업무로 바쁘다. 시스템이 SA의 주 30분만 쓰게 만들면 ROI가 명확하다. 주 3시간을 쓰게 만들면 — 외부 위탁이 더 쌌다. UI의 모든 화면은 "SA가 이 화면에 머무는 시간"으로 평가하라. 가장 빠른 결정 + 충분한 audit trail이 동시에 가능한 단일 화면이 좋은 화면.

2. 5개 화면의 정보 구조

2.1 Queue — 한 화면, 한 작업 리스트

SA가 시스템에 로그인하면 첫 화면. 미처리 case를 P0·P1·P2 순으로 정렬.

▮ Queue — 2026-06-15 09:14
P0 Anchor Cream v2 (SKU-2026-014) · MoS < 100 detected (Phenoxyethanol 0.8%, est. MoS 82) · 2분 전
P1 Niacinamide Toner (SKU-2026-008) · Evaluator 3 retry failed, B.3 narrative score 0.74 · 17분 전
P1 Hydra Cream v1.3 (SKU-2025-201) · Formula changed: +Tocopherol 0.2 → 0.4 · 38분 전
P2 Calming Serum (SKU-2026-011) · Tier 3 confidence "low" for 1 substance (Centella Asiatica Extract) · 1시간 전
DRAFT READY Vitamin C Ampoule (SKU-2026-013) · All checks passed, awaiting signature · 1시간 전
DRAFT READY Cleansing Foam (SKU-2026-010) · All checks passed · 3시간 전

각 row의 디자인 원칙:

  • 제목 한 줄: 제품명·SKU·트리거 사유.
  • P0는 색·아이콘으로 즉시 구별 — 못 보고 지나칠 위험 제거.
  • row 클릭 → 즉시 Case Detail로. 중간 화면 없음.
  • 대기 시간 표시: SA가 의식하게.
  • 완료된 case는 24시간 후 hidden: 시각적 노이즈 제거. audit log에는 영구 보관.

2.2 Case Detail — 한 결정을 위한 모든 정보

P0 case 하나를 열었을 때의 화면. SA가 결정에 필요한 모든 정보가 한 페이지에:

▮ Case · Anchor Cream v2 (SKU-2026-014) · P0

Trigger: Phenoxyethanol 0.8% → SED 0.082 mg/kg bw/day, NOAEL 1000 mg/kg bw/day (SCCS/1575/16) → MoS = 82 (임계 100 미달).

Context: 사내 자체 표준은 leave-on phenoxyethanol ≤ 0.5%. 현 처방 0.8%는 사내 standard 초과.

Tier 3 Data: SCCS Opinion SCCS/1575/16 (Phenoxyethanol, 2016), p.40 §4.3.2. NOAEL 1000 mg/kg bw/day (rat sub-chronic). DA% 25% (in vitro Franz cell). · view PDF anchor

4 Decision Options:

  1. 농도 인하 → 0.5%로 (re-run 자동)
  2. 처방 교체 → Ethylhexylglycerin 단독 검토 의뢰 (R&D 알림)
  3. Species-specific UF 재검토 → human PBPK 데이터 첨부
  4. 자체 시험 — in vivo dermal study 의뢰

Override:

Comment: (필수 입력 — 결정의 사유)

이 화면의 설계 원칙:

  • 버튼이 가짜가 아니다. "농도 인하 0.5%"를 누르면 시스템이 즉시 새 SED·MoS를 자동 재계산하고 narrative를 regenerate.
  • Override 옵션은 항상 존재하지만 reason 필수 — SA가 "법적 책임 영역"에서 자체 판단 가능.
  • 모든 외부 자료는 anchor 링크로 — PDF 페이지·section 직접 점프. RAG가 모은 page_anchor가 여기서 쓰임.
  • comment는 학습 자산. 추후 reference set의 unwritten rule 추출 데이터.

2.3 Decision Recorder — 한 클릭 = 한 줄 audit

SA가 위 화면에서 옵션을 선택한 순간 자동 기록되는 데이터:

class SaDecision(BaseModel):
    decision_id: str
    case_id: str
    sa_id: str
    decided_at: datetime
    trigger_type: Literal["mos_below_threshold", "no_noael",
                          "evaluator_failed", "formula_changed",
                          "low_confidence", "manual_review"]
    trigger_payload: dict  # 트리거 시점의 fact bundle snapshot

    decision: Literal["reduce_concentration", "replace_substance",
                       "refine_uf", "commission_test", "override",
                       "approve_as_is", "reject"]
    decision_params: dict  # {"new_pct": 0.5} 등 구체 파라미터
    sa_comment: str  # 필수, free text

    propagation: list[Propagation]  # 결정 이후 시스템 행동

SA가 한 번 결정하면 시스템이 4가지 propagation을 자동 수행:

Propagation설명
1. Re-run pipelinefact bundle 갱신 후 T1·T2·T3·T4 자동 재실행
2. Update reference rules"phenoxyethanol leave-on ≤ 0.5% 사내 표준" 같은 unwritten rule을 internal_rules.yaml에 자동 후보로 등록 (SA 검토 후 승격)
3. Notify R&D처방 변경이 필요한 결정이면 R&D Slack/이메일로 알림
4. Audit log entry변경 불가능한 append-only log에 한 줄 기록

2.4 Audit Log — append-only, 검색 가능

모든 결정·시스템 자동 행동을 시간순으로 기록. 분기 audit·외부 감독 대응의 핵심 자료.

2026-06-15 09:14:08 · SYSTEM · Trigger P0 · Anchor Cream v2 · MoS=82 < threshold 100
2026-06-15 09:16:23 · SA-001  · Decision: reduce_concentration · phenoxyethanol 0.8→0.5%
2026-06-15 09:16:24 · SYSTEM · Pipeline re-run started · run_id=2026-06-15-091624
2026-06-15 09:17:51 · SYSTEM · MoS recalculated = 131 · all checks passed
2026-06-15 09:18:02 · SYSTEM · Narrative regenerated · evaluator score 0.94
2026-06-15 09:18:05 · SYSTEM · Rule candidate: "phenoxyethanol leave-on internal cap 0.5%"
2026-06-15 09:23:11 · SA-001  · Draft signed · SKU-2026-014 · sig_hash=ab12...

설계 원칙:

  • append-only: row 수정·삭제 불가. 정정이 필요하면 새 row "correction: row #1234"로 기록.
  • SA·SYSTEM·USER 주체 명시: 자동·수동 행동 즉시 구분.
  • 검색·필터: case_id, sa_id, trigger_type, 시간 범위로.
  • 외부 export 1키: CSV / JSON / regulator-friendly PDF.
  • 해시 체인: 각 row hash가 이전 row hash 포함 → 사후 변조 불가능 증명.

2.5 Style Anchor Editor — 사내 화법의 살아있는 문서

SA가 월 1회 30분 들어와 갱신하는 화면. Lesson 0008의 sa_style_anchor.md가 여기서 직접 편집됨.

▮ Style Anchor · v1.3 · last updated 2026-05-30

Mandatory rules

  • 인용 형식: "(SCCS Opinion SCCS/[번호]/[년도])"
  • 결론 화법: "안전 여유가 충분히 확보된 것으로 판단됨"
  • 금지 표현: "안전하다" 단정, "권장"

Preferred patterns

  • 문장 시작: "본 성분 [INCI]은 [농도]%로..." (12회 인용)
  • 전이: "노출 분석 결과...", "한편...", "추가로 고려할 사항으로..."

Rule candidates (pending SA review)

  • Phenoxyethanol leave-on ≤ 0.5% (사내 표준, SA-001 결정 2026-06-15)
  • Tocopherol stabilizer co-use note 권장 (12건 패턴 감지)

이 화면의 핵심: "Rule candidates"가 자동 채워진다. SA의 결정·system의 통계가 후보를 만들고 SA가 1클릭으로 승격하면 generator system prompt에 자동 inject. SA의 시간은 룰 작성이 아니라 "룰 승인"에만 쓰임.

3. 시스템 → SA 알림 채널

Priority채널SLA
P0 (MoS < 임계, NOAEL 부재)Slack DM + 이메일 + UI Queue 상단5분 내 SA 인지

알림 피로 (Alert fatigue)는 시스템 최대 위험 P2 알림을 Slack으로 보내면 SA가 모든 알림을 무시하게 된다. 채널별 정보 가치 ÷ 빈도가 일정 임계 이상이어야 한다. P0이 주 2회 이하라야 Slack DM이 합당. 주 10회 P0이면 트리거 임계가 잘못된 것 — 임계 조정·NOAEL 부재 fallback 강화·자체 시험 의뢰 같은 시스템적 처방이 먼저.

5. Streamlit 첫 화면 — Phase 2 prototype

import streamlit as st
from cpsr import queue_service, case_service, decision_service

st.set_page_config(page_title="CPSR HITL", layout="wide")

if "sa_id" not in st.session_state:
    sa_id = st.text_input("SA ID")
    if sa_id:
        st.session_state["sa_id"] = sa_id
        st.rerun()
    st.stop()

# Queue
st.title("📋 Queue")
cases = queue_service.list_open(sa_id=st.session_state["sa_id"])
for c in cases:
    color = {"P0": "🔴", "P1": "🟡", "P2": "🔵"}.get(c.priority, "⚪")
    if st.button(f"{color} {c.priority} · {c.product_name} · {c.trigger_reason}",
                 key=c.case_id, use_container_width=True):
        st.session_state["open_case"] = c.case_id
        st.rerun()

# Case Detail
if "open_case" in st.session_state:
    case = case_service.get(st.session_state["open_case"])
    st.divider()
    st.subheader(f"Case · {case.product_name} · {case.priority}")
    st.markdown(f"**Trigger**: {case.trigger_summary}")
    st.markdown(f"**Context**: {case.context}")

    # Tier 3 evidence
    with st.expander("Tier 3 evidence (NOAEL source)"):
        st.markdown(case.tier3_evidence_markdown)
        st.link_button("View PDF anchor", case.pdf_anchor_url)

    # 4 decision options
    st.markdown("**Decision options**")
    decision = st.radio("Choose one", case.decision_options, label_visibility="collapsed")
    decision_params = case_service.params_for(decision, case)
    sa_comment = st.text_area("Comment (required)", height=100)

    if st.button("Record decision", type="primary", disabled=not sa_comment.strip()):
        decision_service.record(
            case_id=case.case_id,
            sa_id=st.session_state["sa_id"],
            decision=decision,
            params=decision_params,
            comment=sa_comment,
        )
        st.success("Decision recorded. Pipeline re-run queued.")
        del st.session_state["open_case"]
        st.rerun()

이 ~50줄로 5개 시나리오 중 S1·S2·S3·S4가 작동한다. S5(style anchor)는 별도 페이지. S6(audit)은 DB query로 충분.

6. SA의 결정이 학습 자산이 되는 흐름

UI의 진짜 가치: SA의 모든 결정을 시스템이 학습 데이터로 변환.

SA 행동시스템 학습
"농도 인하 0.5%" 결정 + 이유 "사내 표준"internal_rules.yaml 후보 등록 → 다음 처방에서 자동 사전 점검
"override" 결정 + 이유 "human PBPK 데이터로 UF 50으로 조정"UF 조정 사례 DB 추가 → 유사 성분에서 generator가 "PBPK 기반 UF 조정 사례 있음" 힌트 제공
narrative 수정 제출 (수정된 단락)diff 분석 → "SA가 자주 추가하는 표현" 패턴 통계 → style anchor 후보
특정 NOAEL 출처 선호 (EFSA 대신 CIR 선택)출처 우선순위 학습 → Tier 3에서 후보 정렬 시 SA 선호 반영
"reject" 결정 + 이유reject 패턴 분류 → 미래 trigger 임계 보정 후보
데이터 → 룰 → 시스템 갱신의 cadence 주간: SA 결정 로그를 보고 후보 룰 자동 생성. 월간: SA가 Style Anchor Editor에서 30분 룰 검토·승격. 분기: 승격된 룰을 generator·evaluator·trigger에 정식 반영, regression test로 5건 reference set 평가. SA가 결정한 1건은 한 번에 그치지 않고 같은 패턴을 가진 100건을 미리 처리한다.

8. 외부 감독 대응 — UI가 제공해야 할 것

EU·TFDA·식약처 점검 또는 사내 internal audit 시 UI가 즉시 제공해야 하는 것:

  1. 특정 SKU·revision의 결정 history: 1 SKU → 모든 결정 timeline → 1 PDF export.
  2. 특정 SA의 활동 history: 한 SA가 분기간 처리한 case·결정·서명 list.
  3. 특정 룰의 적용 이력: "phenoxyethanol 0.5% 사내 표준"이 언제부터 적용됐고 어떤 case에 영향 줬는지.
  4. 해시 체인 검증: audit log의 무결성 증명. 단일 명령으로 1년치 chain 검증.
  5. System version + LLM 버전: 각 case 처리 시점의 코드 commit hash + LLM model ID + prompt version.
def export_case_history(sku: str, revision: Optional[str] = None) -> Path:
    """한 SKU의 모든 audit row를 시간순 PDF로 export.

    External audit 응답용. 외부 감독에 보낼 수 있는 형식.
    """
    rows = audit_log.query(sku=sku, revision=revision)
    pdf = pdf_renderer.render(
        title=f"CPSR System Audit Trail · {sku} · rev {revision or 'all'}",
        rows=rows,
        signature_block="This export is hash-chain verified. Chain root: ...",
    )
    return pdf

10. Quiz — 4문항

Q1. Phase 1(CLI + 이메일)에서 SA에게 P0 알림을 어떻게 보낼 것인가? 이메일만으로 충분한가?

답 보기
이메일만으로는 위험하다. Phase 1에서도 P0는 Slack DM 또는 SMS가 필요. 이메일은 SA의 수신함에서 묻힐 수 있다 (P0는 5분 SLA). 권장 조합: (a) Slack DM (개인 채널) — 가장 흔한 사내 도구, (b) 이메일 fallback — Slack 미사용 SA용, (c) UI Queue는 Phase 1에서는 CLI cpsr queue 명령으로 대체. SA가 로컬 터미널을 안 쓴다면 (b)+(c)는 단순 HTML 페이지로 대체 가능 (S3·Netlify·Vercel 정적 호스트). Phase 1의 핵심: P0 알림이 5분 안에 SA의 시선에 도달하는가. 이게 안 되면 UI 없는 phase 1은 운영 불가.

Q2. Audit log에 SA가 narrative를 수정한 경우 "수정 전·후 전체 텍스트"를 모두 저장할 것인가, "diff만" 저장할 것인가?

답 보기
전·후 전체 텍스트를 모두 저장한다. diff는 알고리즘 변경 시 복원 불가 위험 (서로 다른 diff 알고리즘이 다른 결과 산출). 또한 외부 감독이 "수정 전 system 출력이 정확히 무엇이었나"를 요청하면 diff만으로는 답 불가. 비용 우려는 미미 — narrative 1단락 ~500 char, 1년 500건 × 10단락 × 2 (전·후) = ~5MB. 그러나 diff metric은 별도 column에 저장 (char_modification_rate, semantic_change_score)으로 즉시 query 가능. 결론: 전·후 텍스트 + 사전 계산된 diff metrics 둘 다. 무결성을 위한 해시 체인은 텍스트의 hash에 적용.

Q3. SA가 한 case에서 "override — 사내 표준 무시" 결정을 내리고 충분한 이유를 적었다. 시스템은 이 case의 narrative를 어떻게 처리하는가? 시스템이 SA의 override를 어떻게 "기억"하는가?

답 보기
3단계. (1) narrative: SA의 override 이유를 narrative에 자동 inject하지 X — generator는 SA가 별도 입력한 정당화 단락을 그대로 (LLM이 손대지 않고) 본문에 붙임. SA의 자필 텍스트는 LLM이 재구성·축약 금지. (2) audit log: trigger·override·SA 이유·재생성 narrative가 모두 한 case 안에 시간순 기록. 단일 export로 외부 감독에 즉시 제공 가능. (3) 학습: 이 override가 즉시 "사내 표준 변경"으로 일반화되면 X. 같은 트리거·유사 사유의 override가 N건 이상 누적 (예: 3건)되면 "사내 표준 갱신 후보"로 등록 → Style Anchor Editor의 Rule candidates에 노출 → SA가 명시 승격하면 표준 갱신. SA 1명의 1회 override가 시스템 룰을 즉시 바꾸지 X — 인증 SA 합의 + 명시적 승격으로만 룰 변경. 이게 audit·법적 무결성의 기본.

Q4. Streamlit으로 Phase 2를 구현 중. SA 2명이 동시에 다른 case를 열고 결정을 입력하려 한다. Streamlit의 single-process 한계가 문제가 되는가? 어떻게 처리할 것인가?

답 보기
읽기는 동시 가능, 쓰기는 race condition 위험. Streamlit은 사용자별 session_state는 격리하지만, 백엔드 DB write는 공유. (1) 한 case에 한 번에 한 명 lock: case가 열리면 locked_by 필드 set. 다른 SA가 같은 case 열면 "현재 SA-001 검토 중" 표시 + read-only. timeout 30분. (2) Decision Recorder는 idempotent: 같은 decision_id가 두 번 와도 한 번만 적용. (3) DB level transaction: PostgreSQL의 row-level lock 또는 SQLite WAL mode (Phase 2 scale에서는 SQLite도 충분 — SA 동시 사용 ≤ 5명). (4) Streamlit 한계가 boredom 되면 Phase 3으로: Next.js + Supabase Realtime이 multi-user concurrent에 견고. Phase 2에서 SA 동시 사용이 일 5건 이상이면 Phase 3 시기. 그 전까지 SQLite + case lock으로 충분.
