Lesson 0002 · 약 30분

4-Tier 자동화 분류

CPSR의 모든 섹션은 다섯 가지 자동화 등급 중 하나로 분류된다. 이 분류가 시스템 전체의 backbone이다.

이 레슨의 약속 레슨을 마치면 (1) 5개 tier의 정의·예시·실패 모드를 외우고, (2) CPSR Annex I의 모든 섹션을 tier로 분류할 수 있으며, (3) 시스템 아키텍처에서 "어디서 LLM을 쓰고 어디서 안 쓰는가"의 판단 기준이 명확해진다.

1. 왜 분류가 backbone인가

CPSR을 LLM 하나로 한 번에 쓰게 하면 95% acceptance rate는 불가능하다. 이유는 명확하다:

  • 계산 영역 (SED·MoS)에서 LLM은 산수 오류를 낸다. 0% 허용.
  • 인용 영역 (NOAEL)에서 LLM은 fabrication한다. 추적 가능한 grounding 없으면 SA가 거부.
  • 판단 영역 (B.3 Reasoning)에서 LLM은 가치 있는 일을 한다. 하지만 rubric 평가 없으면 일관성 없음.
  • 책임 영역 (B.4 서명)은 법적으로 사람만 가능.

같은 도구로 다른 문제를 풀면 안 된다. 섹션마다 적합한 tier로 라우팅해야 한다. 그 라우팅 테이블이 곧 자동화 분류다.

2. 다섯 가지 Tier

TIER 1 Deterministic Compute

입력이 정해지면 출력이 수식으로 결정. LLM 사용 금지. unit test 100% 커버.

예시 (CPSR)
SED 계산, MoS 계산, retention factor 곱셈, 단위 변환, 농도 합계 100% 검증, aggregate exposure 합산
구현
순수 함수, Python/TypeScript module. compute_sed(amount, concentration, da_pct, retention, bw) -> float
실패 모드
입력 단위 혼동(mg vs μg, % vs ppm), retention factor 잘못 매핑, 체중 기본값(60kg) 미명시. 입력 검증 레이어가 진짜 위험.
검증
SCCS NoG 12판 부록의 example calculation을 unit test로 박제
TIER 2 Structured Data Retrieval

외부의 구조화된 데이터를 그대로 가져옴. LLM 불필요. API/DB 쿼리.

예시 (CPSR)
CosIng에서 INCI 정규화·기능·EU Annex 상태 조회, ECHA REACH 등록 substance ID 매핑, CAS·EC 번호 lookup, 사내 R&D DB에서 처방·안정성 데이터 fetch
구현
API client, 캐시 레이어, 정규화 스키마. CosIng은 공식 API 없음 → 정기 dump 활용 또는 scraping (사내용은 회색지대, 외부용은 라이선스 확인).
실패 모드
INCI 표기 변종(소문자·trailing space·동의어), 비활성 INCI(폐기), regional naming(예: Korea 표준명 ≠ INCI), CAS multi-mapping
검증
known-good fixture 100개로 정규화 정확도 측정. 임계 미달 시 Tier 3로 escalate.
TIER 3 LLM Extraction with Grounding (RAG)

비구조화 1차 자료(PDF·HTML)에서 LLM이 사실을 추출. 출처 anchor 강제. fabrication 0% 목표.

예시 (CPSR)
SCCS Opinion PDF에서 성분 NOAEL/SED 한계/MoS·결론 추출, CIR final report에서 권장 최대 농도 추출, REACH dossier PDF에서 acute/repeated dose 데이터 추출, IFRA Standards에서 향료 카테고리별 한도 추출
구현
구조화 출력(JSON schema with citation field). 인용에 문서 ID + 페이지/문단 anchor + 원문 quote 필수. 추출 후 quote가 원문에 literal match하는지 검증.
실패 모드
(a) Fabrication: 원문에 없는 NOAEL을 그럴듯하게 만들어냄. (b) 단위 변환 오류: mg/kg/day vs μg/kg/day. (c) 종 오인: rat NOAEL을 human에 그대로 적용. (d) 다중 시험 중 보수적 값을 안 고름.
검증
quote literal match + 별도 evaluator agent의 "이 NOAEL이 이 문서에 실제로 있는가" 재확인 + 단위 일치 룰. 정합성 < 95%면 HITL.
TIER 4 LLM Reasoning with Rubric Evaluation

Tier 1~3의 결과를 받아 LLM이 narrative reasoning을 생성. 별도 evaluator가 rubric으로 채점. acceptance 95%의 핵심 격전지.

예시 (CPSR)
B.3 Reasoning(각 성분의 MoS 평가, 상호작용, 데이터 갭 정당화), B.1 Conclusion(전체 안전성 결론), A.4 불순물 평가 narrative, A.8 toxicological profile 요약
구현
Generator agent + Evaluator agent. Generator는 Tier 1~3 결과를 system prompt에 주입, SCCS NoG 12판의 §3-6/§3-7 평가 흐름을 rubric으로 사용. Evaluator는 rubric 점수 + 발견된 결함 목록 산출. 임계 미달 시 retry 또는 HITL.
실패 모드
(a) Boilerplate화: 모든 성분에 동일 문구 → 평가자 거부. (b) MoS < 100인데 "safe" 결론. (c) 데이터 갭을 침묵하고 넘어감. (d) 종간차·민감군(임산부·영아) 미고려. (e) SCCS NoG 표현·용어와 어긋남.
검증
Rubric 정량 점수 + 실제 SA의 historical reports로 만든 reference set 대비 ROUGE/사실일치도 + spot human review.
TIER 5 / HITL Human-in-the-Loop (필수 사람)

법적·과학적 이유로 사람이 반드시 개입. 자동화 절대 불가 영역.

예시 (CPSR)
B.4 자격자 서명·승인(EU Reg Art. 10 자격 요구), Tier 3에서 데이터 갭 감지 시 SA 호출, Tier 4 evaluator 점수 미달 시 SA 호출, 신규 성분(SCCS Opinion·CIR·ECHA 모두 데이터 없음) 처리, 회사 정책 결정(예: MoS = 80인 성분을 농도 조정할지 처방 교체할지)
구현
워크플로우 게이트 + 알림(slack·email·내부 도구). SA가 결정 → 결정 로그 → 시스템이 학습 데이터로 흡수.
실패 모드
(a) Tier 4에서 silent failure → HITL 호출 누락. (b) SA 결정이 시스템에 피드백 안 됨 → 다음 제품에 동일 갭 반복.
검증
HITL 트리거 false negative rate를 명시적 메트릭으로 추적. 시스템이 "안 부른" 케이스 중 사후 SA가 손댄 비율.

3. CPSR Annex I 섹션 → Tier 매핑 (초안)

섹션주요 Tier비고
A.1 Composition (INCI %)T1 + T2입력 검증 + CosIng 정규화. 처방표 자체는 사람 입력(R&D 제공).
A.2 Physico-chem & stabilityT2 + T4실험실 데이터 fetch(T2) + 의미 해석(T4).
A.3 Microbiological qualityT2 + T4Challenge test 결과 + ISO 11930 적합성 판단.
A.4 Impurities & packagingT2 + T3 + T4중금속 한도 비교(T2), 용기 이행물질 자료 추출(T3), 종합 평가(T4).
A.5 Normal & foreseeable useT1 + T4SCCS NoG Table 1 mapping + 사용 시나리오 narrative.
A.6 Product exposureT1전적으로 deterministic. 카테고리·체중·빈도 → mg/kg bw/day.
A.7 Substance exposure (SED)T1T1.dermal absorption 기본값(50%) 사용 시 명시. 측정값 있으면 T3로 추출.
A.8 Toxicological profileT3 + T4RAG로 NOAEL 추출, 종합 narrative 생성. 핵심 위험 영역.
A.9 Undesirable effectsT2 + T5사내 CRM/QA 시스템 fetch. 신제품이면 N/A.
A.10 Other infoT2 + T4HRIPT·임상 데이터 통합.
B.1 ConclusionT4 + T5LLM 초안, SA 최종 확정.
B.2 Labelled warningsT2 + T4Annex III/IV/V/VI 의무 표시 rule lookup + 추가 경고 판단.
B.3 ReasoningT4분량이 가장 크고 가장 어려움. evaluator rubric 필수.
B.4 Assessor credentialsT5영구히 사람.
설계 함정 T3·T4를 한 LLM 호출로 묶지 마라. T3는 "사실 추출"이고 T4는 "판단". 묶으면 fabricated NOAEL 위에 reasoning이 쌓여 acceptance rate가 무너진다. T3 결과는 evaluator로 검증 후 structured fact로 T4에 전달한다.

4. 시스템 아키텍처 스케치

# CPSR Agent System — Tier-routed pipeline INPUT: {formulation_INCI%, product_category, use_scenario} | v [T1] formulation_validator # sum=100%, ranges, units [T2] cosing_normalizer # INCI → canonical + CAS + EU annex [T1] exposure_calculator # RfE * concentration → SED per substance [T2] internal_data_fetcher # R&D, stability, microbio, undesired effects | v [T3] tox_rag # SCCS Opinion + CIR + ECHA → NOAEL per substance └─ evaluator # quote literal match, unit check, species check | v [T1] mos_calculator # MoS = NOAEL / SED, flag <100 | v [T4] reasoning_generator # B.3 narrative, per-substance + global └─ evaluator(rubric) # SCCS NoG-derived rubric [T4] conclusion_generator # B.1 + B.2 warnings | v [T5] HITL gate # SA review + sign B.4 | v OUTPUT: CPSR draft (JSON) + Word/PDF render + audit log

SCCS NoG 12판 §3-6의 평가 흐름과 정확히 일치하게 단계 순서를 잡았다. 검토 시 SA가 따라가는 사고 흐름과 1:1 대응.

5. Quiz — Tier 분류 연습 (4문항)

Q1. "처방의 페녹시에탄올 0.8%가 EU Annex V (preservatives)에 등재된 보존제 한도 1.0% 이내인지 자동 확인" — 어느 tier?

답 보기
T1 (deterministic compute) + T2 (structured retrieval). T2가 Annex V에서 페녹시에탄올 한도(1.0%)를 fetch, T1이 0.8 ≤ 1.0 비교. LLM 불필요. unit test로 100% 커버 가능.

Q2. "처방의 신규 식물추출물(SCCS Opinion·CIR·ECHA 모두 데이터 없음)에 대한 안전성 판단" — 어느 tier?

답 보기
T5 (HITL). 이건 시스템이 "처리"하면 안 된다. T3 RAG가 "데이터 없음" 응답 → 시스템이 SA 호출 → SA가 (a) read-across·QSAR로 정당화, (b) 농도 제한, (c) 처방 교체, (d) 자체 시험 발주 중 선택. 시스템은 결정을 학습 데이터로 흡수.

Q3. "Niacinamide 4% 처방의 MoS 계산"에서 NOAEL 부분과 SED 부분은 각각 어느 tier로 나뉘나?

답 보기
NOAEL: T3 (SCCS·CIR Opinion PDF에서 RAG로 추출 + quote anchor + 단위/종 검증). SED: T1 (face cream RfE 1.54 g/day × 4% × dermal absorption × 1.0 retention / 60kg). MoS = NOAEL ÷ SED: T1 (순수 산수). 결과의 narrative 해석 한 줄: T4. 한 번의 산출에 T1·T3·T4가 모두 들어간다 — tier 라우팅이 왜 backbone인지의 모범 예.

Q4. Tier 4 reasoning_generator가 "이 제품은 안전합니다"라는 결론을 95% 일관되게 내려도 acceptance가 안 나오는 이유 두 가지를 들어라.

답 보기
(1) Boilerplate 문제 — 모든 성분에 같은 문구가 반복되면 SA가 "기계가 썼다"는 신호로 인식하고 신뢰를 잃는다. acceptance는 "그대로 쓸 만하다"는 판단인데, 동일 문구로 50개 성분을 처리한 보고서는 누구라도 손대고 싶어진다. (2) 침묵의 갭 — 데이터 없는 성분, MoS < 100인 성분, 민감군 고려를 언급 없이 넘어가면 SA가 검토 단계에서 발견하고 큰 폭으로 수정한다. rubric에 "갭·예외·민감군 명시"를 평가 항목으로 박는 이유.

6. 다음 단계

이 backbone을 확보했으니, 다음 두 갈래 중 하나:

  1. Tier 1 깊이 파기 — 노출량·MoS 계산 공식 마스터. SED·MoS·retention factor·RfE·체중 기본값을 SCCS NoG 12판 §3-6에서 정확히 박제. unit test 묶음 설계의 baseline.
  2. Tier 3 깊이 파기 — SCCS Opinion 코퍼스 구조 분석. Opinion PDF의 일관 구조(요약·NOAEL·MoS·결론 섹션)를 파악해 RAG schema·prompt를 설계하기 위한 기반.
교사에게 질문 "T2에서 CosIng API 없는 문제는 어떻게 우회하나?", "T3 evaluator가 quote literal match를 강제할 때 PDF OCR 텍스트와 원문 문자 차이는?", "T4 rubric의 SCCS NoG 발췌는 어디까지 정량 추출 가능한가?" — 이런 실무 질문은 다음 lesson을 보다 정확히 만든다.
