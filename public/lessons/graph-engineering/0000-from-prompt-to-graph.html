Lesson 0000 · Graph Engineering

프롬프트 → 컨텍스트 → 하네스 → 루프 → 그래프

요즘 단어가 너무 빨리 바뀐다. 새 유행을 “작별 선언”으로 듣지 말고, 한 층 위에 무엇을 올리는지로 들으면 된다.

오늘의 한 줄 답: 다섯 층은 서로 대체 관계가 아니라 쌓는 관계다. 아래 층이 약하면 위 층은 멋져 보여도 쉽게 무너진다.

0. 왜 이 레슨이 필요한가

타임라인에는 동시에 이런 말이 떠다닌다.

  • “루프 엔지니어링과 작별, 이제 그래프”
  • “루프는 괜찮다. 밑의 폴더(하네스)가 안 짜여 있을 뿐”
  • “graph engineering이 왔다”

셋 다 틀린 말이 아닐 수 있다. 다만 가리키는 층이 다르다. 이 레슨은 그 층을 한 사다리로 고정한다. 참고 인포그래픽의 5단 정의와, harness를 바닥으로 보는 실무 글, 우리가 이미 배운 루프/그래프 논의를 한 줄로 잇는다.

1. 다섯 층 사다리

1 Prompt engineering
한 번 호출에 넣을 지시문을 잘 쓴다. “무엇을 어떻게 답하라.”
2 Context engineering
모델이 무엇을 보게 할지 고른다. 파일, 검색 결과, 메모리, 대화 이력, 토큰 예산.
3 Harness engineering
모델 바깥의 실행 환경. 도구, 권한, 훅, 스킬, MCP, 상태 파일, 정책.
4 Loop engineering
그 환경 위에서 목표→행동→검증→계속/중단을 돌린다. 오래 달리게 하되, 검증 가능하게.
5 Graph engineering
여러 작업/루프/에이전트를 의존·병렬·검증·실패 격리로 연결한다. (그리고 개선 루프끼리의 견제도 그래프다.) 
Prompt     : 한 문장/한 턴의 지시
Context    : 그 턴에 넣는 재료
Harness    : 매번 같은 무대·규칙·도구
Loop       : 무대 위 사이클 (goal → act → verify → memory → again?)
Graph      : 여러 사이클/노드의 관계 (실행 배선 + 견제 배선)
대체 신화 깨기: “프롬프트 끝 → 컨텍스트 → 하네스 → 루프 끝 → 그래프”처럼 아래를 버리는 서술은 거의 항상 과장이다. 그래프 안에도 프롬프트·컨텍스트·하네스·로컬 루프가 그대로 산다.

2. 층마다 “고장 나면 어디를 고치나”

Prompt 고장
지시가 애매, 역할 혼선, 출력 형식 없음 → 문장 수정.
Context 고장
엉뚱한 파일, 과한 히스토리, context rot → 재료 선별·요약·프루닝.
Harness 고장
권한 막힘, 훅 없음, 스킬 미로딩, MCP 과다 → 폴더/설정/도구 배선.
Loop 고장
검증 없이 통과, 목표 표류, 같은 스텝 반복 → goal spec, verify, stop 조건.
Graph 고장
병렬 가능 일을 줄세움, 자기검증만, 핸드오프 뭉개짐, 실패가 전체 중단 → 토폴로지·계약·하드 게이트.

Archive 글의 핵심 문장: The harness is the kitchen. The loop is the recipe. 주방이 없으면 레시피는 공상이고, 레시피 없는 주방은 빈 공간이다. 그리고 순서는 보통 하네스 먼저, 루프 다음이다. 권한이 없으면 루프가 “할 수 있는 척” 날조하기 쉽기 때문이다. Loop and Harness engineering (ArchiveExplorer)

3. 루프 안의 최소 5보 (하네스 위)

실무 루프 글들이 반복하는 최소 세트:

  1. Goal spec — 완료 정의가 디스크에 있다 (에이전트 머리속이 아님)
  2. Plan
  3. Act
  4. Verify — 가능하면 별 컨텍스트/별 노드
  5. Memory / stop — 진행 기록, 계속 여부, 정지 조건

검증을 빼면 “자신 있는 쓰레기”가 다음 입력으로 쌓인다. 상태 파일을 안 쓰면 같은 스텝을 다시 계획하는 Ralph-loop 실패가 난다.

4. 그래프는 한 종류가 아니다

A. 실행 그래프
노드 = 작업(조사A, 조사B, 요약, 테스트).
엣지 = 의존·병렬·라우팅·핸드오프.
질문: 일을 어떻게 펼칠까?
n8n / LangGraph / “루프 작별” 중국어 글이 주로 여기.
B. 개선 루프 그래프
노드 = 루프 역할(최적화, 감시, 감사, 목표 소유).
엣지 = pairing / veto / hierarchy.
질문: 개선이 스스로를 속이지 않게 하려면?
Perez 에세이가 여기. Loop→Graph?

둘 다 graph engineering이라 불릴 수 있다. 혼동하면 “루프를 버려야 하나?”로 간다. 정답에 가까운 말:

버리는 것은 루프가 아니라 만능 단일 루프 / 한 줄 파이프라인으로 모든 걸 처리하려는 습관이다. 그래프 안 로컬 루프는 남는다. 정지 조건 있는 재시도는 여전히 루프다.

5. Aside / PATHY로 꽂아 보기

Prompt     지금 채팅 한 줄 지시 ("STORM 영상 올려")
Context    열린 탭, 첨부, memory, 이전 대화
Harness    Aside + skills + Voicebox + browser + 정책
Loop       길이/더듬 검사 → 재생성 → 자막 off → 재mux
Graph-exec  대본│이미지 병렬 준비 → 합치기 → 업로드
Graph-imp   L1 제작 최적화 + L2 품질 감시 + 앵커(실제 URL/청취)

네가 하던 “하네스 조합이 곧 루프” 감각은, 사실 3층 하네스 위에 4층 루프를 자주 올려둔 상태였다. 이제 5층에서 실행 분기와 견제 루프를 구분하면 된다.

6. 진단 치트 (5질문)

  1. 한 번 호출로 끝나야 하는데 말이 안 먹히나? → Prompt / Context
  2. 매번 같은 도구·권한이 막히나? → Harness
  3. 돌긴 도는데 목표에서 벗어나거나 검증이 약하나? → Loop
  4. 서로 독립인 일이 줄 서 있거나, 한 실패가 전부 죽이나? → 실행 Graph
  5. 지표는 초록인데 현실이 나빠지나? → 개선 Graph (pairing/audit/anchor)
