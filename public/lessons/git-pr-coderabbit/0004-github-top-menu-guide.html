Lesson 0004

GitHub 상단 메뉴 활용법

GitHub 저장소 상단 메뉴는 “코드 보기, 일감 관리, 변경 검토, 자동화 확인, 문서 관리, 보안 확인, 운영 설정”으로 나뉩니다. 처음에는 각 메뉴를 눌렀을 때 어떤 질문에 답하는지만 알면 됩니다.

Code Issues Pull requests Agents Actions Projects Wiki Security and quality Insights Settings

Code

무엇: 저장소의 파일, 폴더, branch, commit history, README를 보는 기본 화면입니다.

언제: 파일 위치를 찾거나, 최신 commit이 무엇인지 보거나, branch를 바꾸어 특정 작업 상태를 확인할 때 씁니다.

RISELAB에서: 지금 화면의 main branch는 공식 상태입니다. PR 작업 branch는 branch dropdown에서 따로 봅니다.

Issues

무엇: 해야 할 일, 버그, 질문, 논의 주제를 ticket처럼 모아두는 곳입니다.

언제: “무엇을 해야 하는가”를 기록하고, 담당자/라벨/우선순위를 붙이고, PR과 연결할 때 씁니다.

주의: Issue는 작업 계획이고 PR은 실제 변경 제안입니다. 둘은 연결될 수 있지만 같은 것은 아닙니다.

Pull requests

무엇: branch의 변경을 main에 합쳐도 되는지 검토하는 공간입니다.

언제: 변경 내용, Files changed, Checks, 리뷰 comment, approval, merge 가능 여부를 볼 때 씁니다.

RISELAB에서: PR 제목/본문은 한국어여야 하고, Minimal Verification 통과와 사람 approval이 필요합니다.

PR에서 보는 탭무엇을 확인하나
Conversation요약, 리뷰 대화, CodeRabbit comment, merge box
Commits이 PR에 올라온 commit 목록
ChecksActions, Vercel, CodeRabbit 같은 자동 검사 결과
Files changed실제로 어떤 줄이 바뀌는지

Agents

무엇: GitHub Copilot cloud agent나 연결된 coding agent 작업을 보는 영역입니다. agent는 repo를 읽고 branch에서 변경을 만들고 PR을 준비할 수 있습니다.

언제: AI에게 issue나 prompt 기반 작업을 맡기거나, agent가 만든 branch/PR을 확인할 때 봅니다.

중요: agent는 작업자입니다. agent가 만든 변경도 PR, Actions, CodeRabbit, 사람 approval을 통과해야 합니다.

Actions

무엇: 테스트, 빌드, 배포 같은 자동 workflow가 실행되는 곳입니다.

언제: PR check가 왜 실패했는지 로그를 보거나, 어떤 workflow가 언제 돌았는지 확인할 때 씁니다.

RISELAB에서: PR Checks workflow의 Minimal Verification이 PR gate의 핵심 자동 검사입니다.

Projects

무엇: issue, PR, draft item을 table/board/roadmap으로 정리하는 작업 관리판입니다.

언제: 여러 작업의 상태, 우선순위, 담당자, 마감일, 로드맵을 보고 싶을 때 씁니다.

활용: “해야 할 일은 Issue, 실제 변경은 PR, 전체 흐름 관리는 Projects”로 나누면 이해가 쉽습니다.

Wiki

무엇: README보다 긴 운영 문서, 사용법, 설계 원칙을 모아두는 문서 공간입니다.

언제: 기능 사용법, 팀 규칙, 운영 절차, 배경 설명을 장기 문서로 남길 때 씁니다.

RISELAB에서: 운영 방식이 바뀌면 관련 wiki와 index도 같이 업데이트하는 규칙이 있습니다.

Security and quality

무엇: secret scanning, code scanning, Dependabot alerts 같은 보안/품질 경고를 보는 곳입니다.

언제: API key가 올라갔는지, 취약한 dependency가 있는지, 코드 보안 경고가 있는지 확인할 때 씁니다.

주의: 보안 경고는 단순 테스트 실패보다 위험도가 높습니다. 임의로 무시하지 말고 원인을 확인해야 합니다.

Insights

무엇: commit 활동, contributor, traffic, forks, pulse 같은 저장소 활동 통계를 보는 곳입니다.

언제: 누가 얼마나 기여했는지, 최근 PR/issue 활동이 어떤지, 방문자/clone 추세를 보고 싶을 때 씁니다.

활용: 운영자가 “프로젝트가 살아 있는지”, “최근 변경이 많았는지”를 빠르게 보는 dashboard처럼 씁니다.

Settings

무엇: 저장소 이름, 권한, branch protection, Actions 권한, secrets, Pages, feature enable/disable 등을 바꾸는 관리자 영역입니다.

언제: PR 승인 필수화, required status check, collaborator 권한, secret 등록, Actions 제한 같은 운영 규칙을 설정할 때 씁니다.

주의: Settings는 저장소 전체 동작을 바꾸는 곳입니다. 특히 branch protection, secrets, Actions 권한, visibility는 변경 전후를 기록해야 합니다.

짧은 확인

PR의 테스트 실패 로그를 자세히 보려면 어느 메뉴를 봐야 할까요?
