Lesson 0010

Trade Intel 페이지 속도 개선 케이스 스터디

실제 PR #7 작업을 예시로, 100개가 넘는 페이지를 어떻게 같은 기준으로 리뷰하고, 병목을 줄이고, GitHub Flow와 100% coverage gate로 안전하게 merge했는지 정리합니다.

한 줄 요약

처음 화면을 열기 전에 다 준비하던 무거운 일을 걷어내고, 꼭 필요한 화면부터 빠르게 보여준 뒤 나머지는 뒤에서 불러오게 바꿨습니다.

이번 작업의 숫자

항목결과쉬운 뜻
대상 페이지102개 route관리자, 영업, 물류, 벤더, 바이어 등 실제 화면 대부분을 같은 조건에서 재봤습니다.
목표 기준50ms 이하각 페이지가 처음 응답을 빠르게 돌려주는지 확인하는 기준입니다.
최종 결과102/102 통과측정한 모든 페이지가 기준 안에 들어왔습니다.
가장 느린 페이지45.2ms가장 느린 페이지도 50ms 아래였습니다.
작업 방식branch → PR → checks → merge바로 main에 넣지 않고 검토 공간을 거쳤습니다.

이 작업은 loop/goal 모드로 진행했다

이번 사례는 한 번 수정하고 끝낸 작업이 아니라, 아래와 같은 루프 프롬프트를 입력하고 goal 모드로 오래 돌린 작업입니다.

Continue optimizing the code for speed. measure page-load performance across every page under the same repeatable test conditions. Continue until every page loads in under 50 ms.

쉬운 말로 바꾸면, “모든 페이지를 같은 조건으로 계속 재고, 50ms를 넘는 페이지가 하나도 없을 때까지 멈추지 말라”는 지시였습니다.

반복 측정

페이지를 한두 개만 본 것이 아니라 전체 route를 반복해서 재며 느린 곳을 계속 찾아냈습니다.

반복 수정

느린 페이지를 고친 뒤 다시 전체 기준으로 재측정했습니다. 하나를 고치면 다른 페이지가 병목으로 드러나는 식이라 여러 차례 반복이 필요했습니다.

긴 실행 시간

이 루프는 5시간 이상 이어졌습니다. 사람이 감으로 “이쯤이면 됐다”고 멈춘 것이 아니라, 목표 숫자가 모두 통과될 때까지 계속 확인했습니다.

가장 큰 병목은 무엇이었나

기존 방식

사용자가 페이지를 열면, 화면을 보여주기 전에 주문 상세, 물류 데이터, 통계, 번역, 정산, 편집기 데이터까지 한 번에 준비하는 흐름이 많았습니다.

쉬운 비유

손님이 매장 문을 여는 순간 계산대, 창고, 장부, 진열대, 택배 송장까지 전부 정리한 뒤에야 입장시키는 방식이었습니다.

문제는 기능 자체가 많다는 점이 아니라, 첫 화면에 꼭 필요하지 않은 일까지 첫 로딩 길목에 몰려 있었다는 점이었습니다.

어떻게 리뷰했나

1. 목록화실제 접근 가능한 페이지 route를 모았습니다.
2. 같은 조건 측정같은 서버, 같은 warmup, 같은 반복 조건으로 시간을 쟀습니다.
3. 느린 화면 표시50ms를 넘거나 가까운 페이지를 먼저 봤습니다.
4. 병목 분리첫 화면에 필요한 일과 나중에 해도 되는 일을 나눴습니다.
5. 수정 후 재측정감으로 판단하지 않고 다시 숫자로 확인했습니다.
6. PR 검증테스트와 리뷰 gate를 통과한 뒤 merge했습니다.

해결 방식

바꾼 것왜 빨라지는가비개발자식 설명
page shell 분리처음 화면 뼈대를 먼저 렌더링합니다.문은 먼저 열고, 세부 서류는 뒤에서 가져옵니다.
client loader/server action 분리무거운 상세 데이터 조회를 첫 응답 뒤로 미룹니다.첫 상담 전에 모든 창고 자료를 다 꺼내지 않습니다.
cache 계층 추가반복 계산과 반복 조회를 줄입니다.같은 답을 매번 새로 계산하지 않고 메모해둡니다.
큰 Supabase 조회 분할한 번에 너무 많은 데이터를 끌어오지 않습니다.트럭 한 대에 다 싣지 않고 필요한 박스부터 가져옵니다.
benchmark 스크립트 추가앞으로도 같은 조건으로 다시 측정할 수 있습니다.체중계를 만든 뒤 매번 같은 체중계로 재는 것과 같습니다.

주의할 점

속도 개선은 “기능을 빼서 빠르게 만드는 일”이 아닙니다. 이번 작업의 핵심은 기능을 없애는 것이 아니라, 처음 로딩에 필요 없는 일을 뒤로 옮기는 것이었습니다.

안전하게 진행한 방법

GitHub Flow

codex/page-loadloop branch에서 작업하고 PR #7을 만든 뒤, main에는 PR merge로만 반영했습니다.

100% coverage gate

pnpm test:coverage가 전체 coverage 100%를 요구하도록 설정했습니다. 속도 개선 중 기존 기능이 조용히 깨지는 것을 막기 위한 장치입니다.

smoke/build/typecheck

주요 흐름 smoke test, TypeScript 검사, build, diff check를 함께 통과시켜 “빠르지만 깨진 상태”가 되지 않게 했습니다.

PR 화면에서 무엇을 봤나

Gate확인한 뜻
Minimal Verification프로젝트가 정한 최소 PR 검증이 통과했습니다.
Vercel배포 미리보기 빌드가 성공했습니다.
CodeRabbitAI 리뷰 gate가 성공 상태였습니다.
Mergeable/CLEANGitHub가 이 PR을 main에 합칠 수 있는 상태라고 판단했습니다.

이 케이스에서 배울 점

  • 성능 문제는 느낌이 아니라 같은 조건의 숫자로 봐야 합니다.
  • 첫 화면에 필요한 일과 나중에 해도 되는 일을 나누면 큰 효과가 납니다.
  • 큰 수정일수록 PR, coverage, smoke test 같은 안전장치를 먼저 세워야 합니다.
  • merge 후에도 로컬 main을 최신으로 맞춰야 다음 작업이 올바른 기준에서 시작됩니다.
