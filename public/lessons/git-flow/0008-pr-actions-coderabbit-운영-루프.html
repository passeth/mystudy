Lesson 0005

PR, Actions, CodeRabbit 운영 루프

PR은 변경을 넣는 버튼이 아니라 검토 공간입니다. GitHub Actions는 기계 테스트를 하고, CodeRabbit은 AI 리뷰 의견을 주고, 사람 reviewer는 최종 approval을 줍니다.

한 줄 모델

branch에서 push → PR 생성 → Actions 테스트 → CodeRabbit 리뷰 → 수정 push 반복 → 사람 approval → merge

운영 루프

1. PR 생성branch 변경을 main에 넣고 싶다고 요청합니다.
2. Checks 시작Actions, Vercel, CodeRabbit 등이 자동 실행됩니다.
3. 실패 확인Actions 로그나 CodeRabbit comment를 읽습니다.
4. 수정 push같은 branch에 fix commit을 올립니다.
5. 재검증Checks가 다시 돕니다.
6. 승인/merge모든 gate 통과 후 사람 approval로 merge합니다.

상태값 읽기

상태쉬운 뜻다음 행동
SUCCESS해당 check가 통과했다.다른 gate가 남았는지 본다.
FAILURE테스트/빌드/검증이 실패했다.로그를 읽고 같은 branch에 수정한다.
PENDING아직 실행 중이다.완료될 때까지 기다린 뒤 결과를 본다.
REVIEW_REQUIRED사람 approval이 필요하다.reviewer에게 리뷰를 요청한다.
BLOCKEDmerge 조건 중 하나가 막고 있다.check, review, unresolved thread를 분리해서 본다.

Actions가 하는 일

저장소에 정의된 workflow를 실행합니다. RISELAB에서는 PR마다 최소 테스트와 Agentation build가 돌아가도록 설정했습니다. 실패하면 workflow log와 PR comment를 봅니다.

CodeRabbit이 하는 일

PR diff를 읽고 요약, 개선 의견, 보안/유지보수 지적을 남깁니다. 단, CodeRabbit 통과가 사람 approval을 대신하지는 않습니다.

많이 하는 착각

  • git push를 하면 자동으로 main에 들어간다고 생각한다. 실제로는 branch가 GitHub에 올라간 것입니다.
  • CodeRabbit SUCCESS를 사람 approval로 생각한다. 둘은 다른 gate입니다.
  • Actions 실패를 사용자가 고쳐야 한다고 생각한다. 일반적인 테스트/빌드 실패는 agent가 로그를 읽고 수정해야 합니다.
  • Settings에서 gate를 끄면 해결이라고 생각한다. gate를 끄는 것은 해결이 아니라 보호를 약하게 만드는 일입니다.

CodeRabbit에서 자주 쓰는 명령

PR comment뜻
@coderabbitai review새 변경만 incremental review 요청
@coderabbitai full reviewPR 전체를 다시 review 요청
@coderabbitai pause자동 review 일시정지
@coderabbitai resume자동 review 재개
@coderabbitai resolve이전 CodeRabbit comment 해결 요청

Agent, Codex, Jules는 어디에 들어가나

Codex, GitHub Agents, Jules는 모두 “작업자” 역할로 이해하면 쉽습니다. 이들이 코드를 고치더라도 결과는 branch와 PR로 올라와야 하고, Actions/CodeRabbit/사람 approval gate를 통과해야 합니다.

짧은 확인

CodeRabbit은 통과했지만 REVIEW_REQUIRED가 남아 있습니다. 지금 필요한 것은 무엇일까요?
